一、赛题背景
大语言模型（LLM）正从“能对话”向“能做事”演进。在科研辅助、深度学习、商业调研等高阶知识工作场景中，用户提出的问题往往具有多跳推理、信息验证、工具依赖等特征——仅靠单轮问答无法满足真实需求。

为此，业界提出了 Research Agent 的概念：一个能够自主规划子任务、调用外部工具（如联网搜索、代码执行、知识库检索）、整合多源证据并生成可核查答案的智能体,强调真实世界问题解决能力，要求模型具备推理、浏览、工具调用与答案归一化等综合技能。例如通义团队的DeepResearch。

然而，构建一个工程可用、稳定部署、效果可复现的 Agent 仍面临巨大挑战：

如何在不微调模型的前提下，通过Agent工程提升任务完成率？

如何将工具调用、中间推理、证据整合等环节模块化、可观测、可调试？

如何将整个 Agent 打包为标准 HTTP 服务，供自动化评测系统调用？

本赛题要求参赛者在 阿里云 PAI-LangStudio 平台上，基于 Qwen 系列基础模型，构建一个端到端可部署的 Research Agent，并通过 EAS 服务暴露标准 API。目标是考察选手在Agent 规划 + 工具集成 + 工程落地三位一体的综合能力。

二、赛题设计目标
引导参赛者掌握 PAI-LangStudio 应用开发全流程。

在禁止模型微调的约束下，考察选手如何通过工具策略、提示词优化等方式，最大化基础模型在复杂任务上的表现。

建立可量化、可复现、可审计的评测机制，推动 Agent 架构创新与工程实践标准化。

三、赛题具体内容与评分标准
3.1 任务要求
任务概述
参赛者需在 PAI-LangStudio （工作流模式/代码模式）中开发一个 Research Agent，并将其部署为 PAI-EAS HTTP 服务。该服务需能接收自然语言问题，返回文本形式的答案，且仅依赖允许的模型与工具完成任务。

赛事分为两阶段：

阶段一：提供 100 道公开问题作为验证集，参赛者在此数据上调试 Agent，提交答案，排名前60%的队伍进入下一阶段，此时需要提交 Agent 服务和源码，后续不可更新。

阶段二：使用 100 道非公开问题作为测试集，对参赛者的 Agent 服务打分，得出最终排名。

评测流程

第二阶段

1.评测脚本持有 100 道内部测试集；

2.时间窗口 3 月 9 号到 3 月 13 号；

3.对每个问题，向参赛者 EAS 服务发送 POST 请求（带 Bearer Token，流式请求）；

输入示例：单个自然语言问题，通过 HTTP POST 请求传入，字段名为 question。
curl -X POST \
     -H "Authorization: Bearer <your_token>" \
     -H "Content-Type: application/json" \
     -H "Accept: text/event-stream" \
     -d '{"question": "Where is the capital of France?"}' \

3.4 赛题评分标准
评分指标：正确率

Score = Correct / N

N：评测题目总数（ 100）

Correct：在评测窗口内判定为正确的题目数

其中 Correct 为归一化后与参考答案完全一致的题目数。

若发现硬编码、模型替换、第三方 Agent API 调用等违规行为，该题得分为 0，严重者取消资格。

正确率相同情况下，按照执行耗时从低到高进行排名。

五、竞赛公平性与开放性说明
为确保公平、可审计、可复现，设定以下强制约束：

（1）模型与数据红线
禁止任何形式的模型微调（包括 LoRA、Adapter、自训练权重等）。

仅允许使用以下模型服务：

通过 PAI Model Gallery 部署的 Qwen 基础模型；

通过 阿里云百炼 提供的 Qwen 系列模型服务（禁止启用搜索、网页抓取等工具）。

禁止在提示词、Python 节点、配置文件中硬编码评测题答案（包括哈希映射、关键词触发返回固定答案等）；

允许使用阿里云 IQS、SerpAPI、Bing 等任何不包含 Agent 能力的搜索引擎、允许使用 MCP。

禁止调用任何第三方 Agent 服务，所有 Agent 逻辑必须在 LangStudio 项目包中实现。

（2）项目审计机制
评委会将使用自动化脚本扫描项目配置与代码，检测硬编码、外部调用等违规行为；

请保证阶段二中，EAS 服务是通过LangStudio 项目包部署而来，赛事主办方会做检查，如果发现 EAS 服务中有不属于项目包的逻辑，取消成绩；
